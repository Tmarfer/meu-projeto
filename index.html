Introdução ao Microsoft Fabric

Visão geral da plataforma unificada da Microsoft para análise de dados — o que é, o que não é, e como ela se posiciona como ferramenta oficial do TCU.

👨‍🏫 Prof. Eduardo
⏱️ Módulo 1 de 5
🏷️ Introdução & Visão Geral
🌐

O que é o Microsoft Fabric?

O Fabric é uma plataforma unificada — um "guarda-chuva" ou "canivete suíço" — que consolida diversas soluções de análise de dados da Microsoft que antes eram isoladas: Data Factory, Power BI, Azure Synapse, etc.
ℹ️
Relação com o Power BI

O Power BI é parte do Fabric. Ao clicar no ícone do Power BI no canto inferior esquerdo, você alterna para a interface Fabric. Muitos usuários já usavam o Fabric há cerca de 3 anos sem perceber.

✅
Ferramenta oficial do TCU

O Fabric é a plataforma oficial de análise de dados do TCU, escolhida com base no Quadrante Mágico do Gartner, compatibilidade 100% com Management Studio e custo-benefício.

⬡

Os Três Pilares Estruturantes do Fabric

☁️ OneLake

Storage em nuvem dedicado a dados analíticos (estruturados e não estruturados), análogo ao OneDrive, com espaço ilimitado compartilhado por todos os usuários do Fabric.

⚡ SQL + Spark

O Fabric suporta ambas as linguagens. Spark é recomendado por ser mais rápido, eficiente e distribuído, mas o SQL continua totalmente compatível.

📄 Formato Delta Parquet

Padrão aberto (não proprietário) para armazenamento de dados. Diferente do formato binário do SQL Server ou Oracle, permite que arquivos sejam lidos fora do Fabric por qualquer software compatível.

🚫

O que o Fabric NÃO é

🚫
Não é um servidor de aplicação

Não foi projetado para publicar aplicações com interface visual para entrada de dados (ex.: substituir Apex para cadastros ou Kubernetes para publicação de apps com front-end). Exceção: Fabric Apps (lançado em preview junho/2025) pode cobrir casos de entrada de dados.

🚫
Não é sinônimo de LabContas

O LabContas roda dentro do Fabric, mas o Fabric é muito mais amplo. E-Dados e EPP também devem migrar para ele no futuro.

✅
O Fabric substitui o Kubernetes para ETL

Quando o Kubernetes é usado apenas para ETL ou análise de dados em Python (sem interface visual), o Fabric é o substituto adequado.

⚖️

Fabric vs Databricks

Critério Microsoft Fabric Databricks
Propósito Plataforma unificada de analytics Especializado em Spark pesado
Uso no TCU Análise, notebooks, Power BI, LabContas ETLs pesados do CEPIM e EPP

💡
Regra prática

A escolha entre Fabric e Databricks é baseada em análise de custo. ETLs muito pesados que sobrecarregariam a capacidade do Fabric vão para Databricks — ex.: auditoria com ~1TB roda direto no Fabric; ETLs da Receita Federal vão para Databricks.

🗺️

Roadmap de Migração para a Nuvem

Situação Atual

Dado carregado no SQL Server on-premises → copiado para a nuvem (Fabric) → transformado em Parquet → disponibilizado para análise.

Próxima Fase (~1,5 ano)

Dado entrará diretamente na nuvem. Uma cópia será espelhada para o SQL Server on-premises durante o período de transição.

Estado Final

SQL Server on-premises e PowerCenter serão desligados. Única URL de conexão válida será a do Fabric na nuvem.

E-Dados (Oracle) — Caso Especial

Não pode ser totalmente substituído ainda pois aplicações Apex dependem dele. Arquitetura futura: Databricks lê dado de produção → salva em Parquet no Fabric → exporta para instância Oracle para consumo pelo Apex.

🔐

Segurança e Autenticação

⚠️
Login e senha não funcionam mais

Conexão via login e senha (padrão atual do SQL Server) é insegura e não funciona com o Fabric. O acesso exige MFA/2FA.

✅
Sem VPN

Com MFA, o acesso pode ser feito de qualquer lugar, sem necessidade de VPN ou desktop virtual.

🐍
Python: usar az login

Códigos Python que usam login/senha embutidos não funcionarão mais. Solução: executar previamente o comando az login para autenticação MFA antes de rodar o código Python.

🔮
Próximo passo: login condicional

Acesso restrito a máquinas homologadas pelo tribunal (Defender + Intune configurados) como próxima camada de segurança.

Workspaces — Conceito e Organização

Tudo no Fabric gira em torno de Workspaces. Todo item criado, salvo ou executado pertence a um workspace. Cada workspace é um servidor de banco de dados isolado, com URL de conexão própria, engine, dados e permissionamentos independentes.
🏢
Estratégias de Organização

• Um workspace único para a unidade, com pastas por projeto
• Um workspace dedicado por projeto/fiscalização (recomendação do Eduardo)

👤
"Meu Workspace" Pessoal

Acessível somente pelo próprio usuário, sem possibilidade de compartilhar. Ideal para testes e experimentos.

⚠️
Pastas não têm permissão

Dentro do workspace, a organização é feita por pastas, mas não é possível dar permissão a nível de pasta — apenas a nível de workspace ou de objeto específico.

🏛️
Workspaces do Treinamento

• Treinamento Fabric (ícone verde): gabarito — participantes são Viewers
• Treinamento Fabric Exercícios (ícone cinza): área de exercícios — participantes são Members
• LabContas Nuvem: bases oficiais — acesso individual por perfil SIGA

🖥️ Capacidade F64

64 CPUs na nuvem. Escala automaticamente para 128 ao atingir 100%. Custo: R$ 48k/mês. O SQL Server on-premises tem 128 cores por R$ 64k/mês em licença.

Mecanismo que permite consumo acima de 100% da capacidade por tempo limitado, com cobrança posterior — analogia ao cheque especial bancário.

🔄 Escalabilidade

A nuvem permite reescalonamento dinâmico de capacidade conforme a demanda, pagando proporcionalmente pelo uso.

Formato Delta Parquet vs SQL Database

✅
Padrão no TCU: Delta Parquet

O TCU optou por usar apenas bancos Delta Parquet para incentivar a migração para Spark. Lake House, Warehouse e Event House armazenam dados nesse formato aberto.

⚠️
SQL Database: evitar no TCU

O SQL Database armazena em formato binário (similar ao SQL Server). Não suporta Spark e prende o usuário no paradigma SQL. Por isso, o TCU optou por não utilizá-lo como padrão.

Integração com Git / Versionamento

  • O que é versionado: notebooks, estruturas de banco, pipelines e outros artefatos. Os dados NÃO são versionados pelo Git — apenas estrutura e código.
  • Salvamento automático: ocorre em tempo real no OneLake. O commit para o Git é uma ação separada e opcional.
  • Colaboração simultânea: múltiplos usuários editando o mesmo notebook já é nativo. O Git serve como backup e controle de versão.
  • Vinculação sempre na raiz: pastas não podem ser associadas individualmente a repositórios — a vinculação é sempre no nível do workspace.
  • Branches por projeto: workspaces com múltiplos projetos e equipes podem usar branches para evitar conflitos.
    • ⚠️
    Exclusão de workspace: 60–90 dias

    Workspace sem vínculo ao Git fica na lixeira por 90 dias após exclusão; com Git, pode ser restaurado a qualquer momento. Após o prazo, perde-se tudo: notebooks, bancos e dados do OneLake.

    🗺️

    Catálogo do OneLake & OneLake Explorer

    📂
    Catálogo do OneLake

    Interface tipo Windows Explorer que lista todos os artefatos e bases de dados de todos os workspaces acessíveis. Permite filtrar por domínio, workspace e tipo de objeto.

    💻
    OneLake Explorer (app desktop)

    Aplicativo que funciona como o OneDrive, exibindo todos os workspaces como pastas no sistema de arquivos local. Permite copiar/colar CSV para o OneLake e baixar arquivos Parquet.

    ⚠️
    Cuidado ao apagar arquivos

    Pelo OneLake Explorer é possível apagar arquivos Parquet acidentalmente, destruindo a base de dados na nuvem.

    Tipos de Armazenamento de Dados

    Os diferentes bancos de dados disponíveis no Fabric — Lake House, Warehouse, SQL Database e Event House — e os conceitos de Spark, PySpark, Parquet e Endpoint SQL.

    👨‍🏫 Prof. Eduardo
    ⏱️ Módulo 3 de 5
    🏷️ Armazenamento & Tecnologias
    🗄️

    Os Bancos de Dados do Fabric

    BancoEntrada de DadosLeituraFormatoVisão Files
    Lake House Spark (obrigatório) Spark + SQL Parquet / Delta Sim
    Warehouse SQL (obrigatório) SQL + Spark (só leitura) Parquet (interno) Não
    SQL Database SQL SQL + Spark (via réplica) Binário + Réplica Parquet Não
    Event House Tempo real / Streaming KQL / SQL Índices temporais Não
    📚

    Conceitos Técnicos Fundamentais

    ⚡ Spark

    Engine de processamento distribuído — não é uma linguagem. Aceita Python, R, Scala e SQL como linguagens de comando. 10x mais rápido que SQL para joins e agregações massivas.

    🐍 PySpark

    Python + bibliotecas que conversam com o Spark para processamento distribuído. Substitui o PowerCenter na nuvem para ETL. Usar Python puro (sem Spark) é desperdício — só usa 1 nó do cluster.

    📄 Parquet

    Formato de arquivo que segmenta e particiona tabelas em múltiplos arquivos comprimidos para processamento paralelo. É um formato aberto e portável.

    📦 Delta Table

    Parquet + arquivos de log de transação + metadados. É o formato correto para armazenamento de dados tabulados no Fabric. Aberto e não proprietário.

    🔌 Endpoint SQL

    Camada SQL criada pelo Fabric sobre arquivos Parquet para permitir consultas SQL sem usar Spark. O endereço do endpoint é do workspace (não do banco individual). Todos os bancos (Lake House, Warehouse, SQL Database) possuem endpoint SQL.

    🏠

    Lake House — Data Lake com SQL

    O Lake House possui duas visões distintas: Files (sistema de arquivos — aceita qualquer tipo de arquivo) e Tables (interface SQL). A regra de ouro: SQL só lê — toda criação e inserção deve ser feita via Spark.
    🚫
    CREATE TABLE no Lake House via SQL = erro

    Ao tentar criar tabela via SQL no Lake House, retorna erro "create table was denied". Quando você faz upload de CSV pela interface, por debaixo dos panos está rodando um job Spark.

    ✅
    Exercício: Criando um Lake House

    Nomenclatura padronizada: login_lh_teste. Upload de CSV → seção "Files" → converter para tabela Parquet em "Tables". Consultas SQL via endpoint de análise SQL.

    🏭

    Warehouse — SQL Robusto

    O Warehouse é o inverso do Lake House: SQL lê e grava, Spark apenas lê. Não possui partição Files — toda entrada de dados deve ser via SQL, pipeline (Data Factory) ou notebook com conexão SQL.
    ⚠️
    Não há upload direto no Warehouse

    Warehouse não tem partição Files, não aceita upload de arquivos. Para carregar CSV: use SQL (insert), pipeline ou notebook com conexão SQL.

    💡
    Notebook para carga em Warehouse

    Crie o notebook no workspace (não dentro do Warehouse). O notebook lê CSV da nuvem e insere via SQL no Warehouse.

    ⚡

    Event House — Dados em Tempo Real

    ℹ️
    Banco otimizado para streaming

    Otimizado para registros em tempo real, com índices temporais. Útil para logs e eventos que chegam continuamente (ex.: monitoramento, telemetria, auditoria em tempo real).

    📝

    Boas Práticas e Pontos Críticos

    ⚠️
    Padronização de nomes — CRÍTICO

    Use apenas letras, números e underscore. Evite traços, acentos, espaços e caracteres especiais. O Spark interpreta traço como sinal de menos, causando erros. Isso é incompatível com Windows também.

    ⚠️
    Ordenação case-sensitive

    Por padrão, o Fabric cria workspaces diferenciando maiúsculas/minúsculas. O Cepim sempre altera para case-insensitive. Após a criação do banco não é possível mudar essa configuração.

    🚫
    Python puro no Fabric = desperdício

    Usar Python sem bibliotecas Spark usa apenas 1 nó do cluster. Sempre use PySpark para processamento distribuído.

    🚫
    Acesso remoto ao Spark: impossível da máquina local

    De máquina local, mesmo com PySpark instalado, não é possível acessar o cluster. O acesso remoto usa apenas endpoint SQL. Para usar Spark, execute notebooks dentro do Fabric.

    ⚠️
    Bug Management Studio 22.0.0

    Versão instalada no desktop virtual novo tem bug ao conectar em workspaces onde o usuário tem permissão apenas nos bancos. Atualize para 22.1+ ou use o desktop antigo.

    📸
    Snapshot é responsabilidade do usuário

    A Microsoft faz backup contra falhas, mas snapshots para restaurar cargas corrompidas devem ser criados pelo usuário.

    Aprofundamento: Lake House, Warehouse e SQL Database

    Diferenças práticas entre os três tipos, Shortcuts para integração de dados, conexão via Management Studio e estratégias de data quality.

    👨‍🏫 Prof. Eduardo
    ⏱️ Módulo 4 de 5
    🏷️ Prático & Integração
    📋

    As Regras de Ouro

    🏠 Lake House

    SQL → só lê
    Spark → cria e grava

    Para criar tabelas no Lake House é obrigatório usar Spark.

    🏭 Warehouse

    SQL → lê e grava
    Spark → só lê

    Toda entrada de dados no Warehouse é via SQL. Sem partição Files.

    💾 SQL Database

    SQL → lê e grava
    Spark → via réplica Parquet

    Permite índices personalizados. Grava em binário + réplica Parquet habilitada por padrão.

    ☁️ OneLake

    Storage único e transversal a todos os workspaces. A separação por workspace é apenas lógica quando se acessa via SQL — fisicamente, tudo está no mesmo lugar.

    🔗

    Shortcuts (Atalhos) — Integração entre Fontes

    Um Shortcut é um link criado dentro da partição Files de um Lake House para referenciar dados externos — de outro workspace, SharePoint, OneDrive, AWS, Google Cloud ou Azure Storage — sem duplicar os dados.
    ⚠️
    Shortcuts: apenas no Lake House

    Shortcuts só podem ser criados na partição Files do Lake House (visão de arquivos). Warehouse e SQL Database não suportam shortcuts por não terem partição Files.

    🚫
    Erro comum: criar shortcut na visão SQL

    Muitos tentam criar shortcut quando estão na visão SQL do Lake House. O shortcut só pode ser criado na visão Files.

    ⚠️
    Atalho para esquema inteiro cria dbo_1, dbo_2...

    Ao criar shortcut para esquema inteiro, o sistema cria dbo_1, dbo_2, etc. Recomenda-se criar atalhos por tabela individual.

  • Passo a passo para criar Shortcut para SharePoint:
    Lake House → partição Files → botão direito → "Novo Atalho" → escolher fonte (SharePoint, OneDrive, OneLake, AWS, etc.) → selecionar a pasta/tabela.
  • Exemplo prático: Após criar shortcut para SharePoint do treinamento, os CSVs ficam acessíveis dentro dos Files. Carga do arquivo consulta_candidato.csv do TSE com separador ponto-e-vírgula — o sistema faz tipagem automática.
  • Cruzamento cross-workspace: Criar shortcut para tabela de outro workspace → referenciar no seu warehouse → fazer JOIN SQL normalmente.
    • 🔌

    Conexão via Management Studio

  • Obtenha o endereço do endpoint SQL: acesse o banco → engrenagem → "SQL Endpoint".
  • Abra o Management Studio e conecte usando o endereço copiado com autenticação MFA/2FA.
  • É possível conectar múltiplos workspaces simultaneamente — cada um aparece como servidor separado.
    • ℹ️
    Endereço do endpoint é do workspace, não do banco

    O endereço SQL é do workspace inteiro. Qualquer banco do workspace retorna o mesmo endereço de conexão.

    💻
    Alternativa: VS Code com extensão SQL Server

    Visual Studio Code com a extensão SQL Server da Microsoft permite acesso remoto ao Fabric de forma mais leve que o Management Studio.

    ⚠️
    Workspace único = visão poluída

    Trabalhar com workspace único para múltiplos projetos dificulta a gestão. Ao conectar via Management Studio, você vê todos os bancos de todos os projetos/alunos misturados.

    ⚖️

    Quando usar Spark vs SQL

    TarefaFerramenta idealMotivo
    Joins e agregações massivasSpark10x mais rápido, processamento distribuído
    Busca de registro individualSQLÍndices específicos por coluna
    ETL em larga escalaPySparkSubstitui PowerCenter na nuvem

    ✅

    Estratégias de Data Quality

    🥉 Abordagem Bronze → Silver

    Carga inicial como varchar (sem tipagem, aceita tudo) — camada Bronze. Depois, ETL de tipagem + validação produz a camada Silver já com dados corretos.

    ⚡ Tipagem Direta + Log de Erros

    Tipa diretamente na carga, mas captura e registra os erros de conversão em tabela de log separada para investigação posterior.

    💡
    Dados devem estar na nuvem

    Notebooks rodando no Fabric não conseguem ler CSV da máquina local. É necessário fazer upload para OneDrive, SharePoint ou partição Files de um Lake House antes de processar.

    ⚠️
    Custo de rede entre nuvens

    Se dados estiverem em outra nuvem (AWS, Google), haverá penalidade grande de performance ao ler do Fabric. O ideal é manter dados na Azure para conexão local rápida.

    Consultas SQL e Integração de Dados

    Três formas de consultar dados no Fabric, catálogo do OneLake, itens endossados, cruzamento cross-workspace com Shortcuts, Notebooks SQL e uso do Copilot.

    👨‍🏫 Prof. Eduardo
    ⏱️ Módulo 5 de 5
    🏷️ Consultas & Notebooks
    🔍

    Três Formas de Consulta SQL no Fabric

  • Interface Web do Endpoint SQL
    Acesso direto pelo navegador ao endpoint SQL do lakehouse ou warehouse. Ideal para consultas rápidas ad hoc. Suporta Copilot para geração assistida de queries.
  • Management Studio (conexão externa)
    Ferramenta desktop familiar para quem usa SQL Server. Conecte usando o endereço do endpoint SQL com autenticação MFA. Permite conectar múltiplos workspaces simultaneamente.
  • Notebooks SQL
    A forma mais poderosa para fluxos de trabalho documentados. Permite agendar execução, salva histórico automaticamente, suporta visualizações e gráficos inline.
    • 📖

    Catálogo do OneLake e Itens Endossados

    O Catálogo do OneLake é o elemento centralizado que lista todos os bancos publicados no Fabric — essencial para o Data Mesh funcionar. Permite encontrar bases mesmo sem ser Viewer no workspace, bastando ter permissão de leitura na base específica.
    🏅
    Itens Endossados

    Bancos marcados como oficiais do tribunal com carimbo de qualidade. Apenas administradores podem endossar itens. Filtrar por "Itens Endossados" é a forma mais rápida de encontrar bases do LabContas sem precisar ter viewer no workspace.

    🚫
    Bases ocultas sem permissão

    Se você não tiver permissão de read na base ou viewer no workspace, a base não aparece no catálogo. Use o sistema Silk para consultar bases disponíveis.

    🌐
    Data Mesh

    Arquitetura onde dados compartilhados podem estar distribuídos em múltiplos workspaces, não centralizados em um único local. O Catálogo do OneLake e os Itens Endossados são os mecanismos de governança que viabilizam essa arquitetura.

    🐍 Bibliotecas Python

    Três formas de adicionar:
    1. Padrão (já incluídas)
    2. pip install (bibliotecas exóticas)
    3. Ambiente customizado pré-configurado

    Notebooks salvam resultados automaticamente

    As 10 Lições Mais Importantes

  • Workspace é a unidade fundamental do Fabric — define permissões, CPU, ordenação e conexão Git. Recomenda-se criar um workspace por projeto/auditoria para controle granular.
  • Três tipos principais de bancos estruturados: Lake House (entrada Spark, leitura Spark/SQL), Warehouse (entrada SQL, leitura SQL/Spark) e SQL Database (SQL puro com opção Parquet).
  • Spark vs SQL: Spark é 10x mais rápido para joins e agregações massivas. SQL é melhor para busca de registro específico com índices. Nunca use Python puro — sempre PySpark.
  • PySpark substitui o PowerCenter na nuvem — Python + bibliotecas Spark para ETL distribuído gravando em Parquet. O código é o mesmo no Fabric e no Databricks.
  • Acesso remoto sempre usa endpoint SQL — para processamento Spark real, execute notebooks dentro do Fabric. De máquina local, mesmo com PySpark instalado, não acessa o cluster.
  • Permissões em dois níveis: workspace (4 perfis) ou banco a banco. Viewer no workspace = acesso a TUDO. Para dados restritos, conceda por banco.
  • Padronização crítica: apenas letras, números e underscore. Traços, acentos e caracteres especiais causam erros no Spark e incompatibilidade com Windows.
  • Shortcuts (Atalhos) só no Lake House (partição Files). Permitem cruzar dados entre workspaces sem duplicação. Joins SQL são limitados ao mesmo workspace — para cruzar, use shortcut.
  • Catálogo do OneLake + Itens Endossados são a governança do Data Mesh — permitem descobrir e acessar bases de outros workspaces sem ser viewer neles.
  • Notebooks SQL são a forma mais completa de consultar dados — documentam o fluxo, permitem agendamento, mas atenção: salvam resultados automaticamente, o que pode expor dados sensíveis no Git.

    Armadilhas a Evitar

    🚫
    Traço no nome dos objetos

    O Spark interpreta traço como operador de subtração. Use apenas underscore.

    🚫
    Case-sensitive após criação

    Não é possível alterar a ordenação do banco após sua criação. Defina case-insensitive na criação.

    🚫
    Viewer no workspace = acesso total

    Para compartilhar com acesso restrito, dê permissão diretamente no banco/objeto.

    🚫
    Notebook SQL + GitHub + dados sensíveis

    Notebooks salvam resultados (até 10k linhas). Se o workspace estiver conectado ao GitHub, dados como CPF podem ser expostos.

    🚫
    CREATE TABLE no Lake House via SQL

    Não funciona. Use Spark para criar tabelas no Lake House.

    🚫
    Shortcut na visão SQL do Lake House

    Shortcuts só podem ser criados na visão Files, nunca na visão SQL.

    ✅

    Boas Práticas Consolidadas

    • Criar um workspace por projeto/auditoria
    • Usar apenas letras, números e underscore nos nomes
    • Configurar case-insensitive na criação do banco
    • Fazer upload de arquivos para SharePoint/OneDrive/Files antes de processar
    • Criar snapshots manuais antes de cargas importantes
    • Conectar o workspace ao GitHub para versionamento de notebooks e estrutura
    • Usar PySpark (nunca Python puro) para ETL no Fabric
    • Criar Shortcuts por tabela individual (não por esquema inteiro)
    • Endossar itens oficiais no catálogo para implementar governança
    • Usar az login antes de scripts Python que acessam o Fabric remotamente
    • Manter dados na Azure para evitar penalidade de rede cross-cloud
